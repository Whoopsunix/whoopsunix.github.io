DevSecOps

0x00 总结​

简单来说就是在 DevOps （开发、运营）的基础上添加了安全需求，在保障应用交付速度的同时更早的、有意识的将安全性需求融入 SDLC 中，强调软件开发交付过程中的安全性。具体表现在 CI/CD 流程中集成应用安全测试 (AST) 工具，比较成熟的几种 AST 产品：

  1. 静态应用安全测试 (SAST)
  2. 软件组件分析 (SCA)
  3. 交互式应用安全测试 (IAST)
  4. 动态分析测试 (DAST)

实际 DevSecOps 的实施是有难度的，不只是引入安全产品就符合要求，重点是安全方面的运营能力，事件响应、安全监控、漏洞管理决定了持续安全的效果，这对开发人员有很大的要求，所以也对应着 DevSecOps 的两个核心原则：安全左移、协同开发。

0x01 基础概念​

CI/CD​

CI/CD 是软件开发的一种自动化实践，持续集成（Continuous Integration）和持续交付/持续部署（Continuous Delivery/Continuous Deployment）的组合，直接引用 GPT 的介绍。

  1. 持续集成（Continuous Integration，CI）： 持续集成是指开发者将代码频繁地提交到共享仓库，并通过自动化工具进行构建和测试。这样可以及时地发现和解决代码集成问题，确保团队的代码始终保持可用和稳定。
  2. 持续交付/持续部署（Continuous Delivery/Continuous Deployment，CD）： 持续交付和持续部署是指自动化地将经过测试的代码变更部署到生产环境中。持续交付要求软件在任何时候都可以被部署到生产环境，而持续部署则更进一步，要求代码通过自动化流程直接部署到生产环境中，从而实现快速、可靠地交付软件。

SDLC 软件开发生命周期​

SDLC 定义了软件开发从需求分析、设计、开发、测试、部署到维护和更新的整个过程，SDLC 通常包括多个阶段和活动，如瀑布模型、迭代模型、敏捷开发等，每个阶段都有特定的目标和活动，以确保软件项目的成功交付。

  1. 瀑布模型（Waterfall Model）： 瀑布模型是SDLC中最传统的一种开发方法，它将软件开发过程分解为线性的阶段，每个阶段依次执行，且在进入下一阶段之前必须完成上一阶段。这些阶段通常包括需求分析、系统设计、实现、测试、部署和维护。瀑布模型适用于需求稳定、项目较小且对风险较低的情况，但缺乏灵活性和反馈机制，不适合对需求变化频繁的项目。
  2. 迭代模型（Iterative Model）： 迭代模型将软件开发过程划分为一系列短期的迭代周期，在每个迭代周期中，团队完成一部分功能，并根据用户反馈和需求变化进行调整和改进。迭代模型强调快速交付、持续改进和迭代开发，有助于更好地应对需求变化和风险管理，但可能会增加项目管理的复杂性。
  3. 敏捷开发（Agile Development）： 敏捷开发是一种灵活的、迭代的开发方法，强调快速交付、持续反馈和团队协作。敏捷开发将软件开发过程分解为多个短期的迭代周期（称为Sprint），每个迭代周期通常持续2到4周，团队在每个迭代周期中完成一部分功能，并根据用户反馈进行调整和改进。敏捷开发鼓励频繁的交付、灵活的需求管理和自组织的团队，有助于提高项目的透明度、质量和客户满意度。
